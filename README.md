# 开发计划


### 项目结构：

``` bash
bill-management/
├── api/                  // 标准布局：API 相关（接口定义、Swagger 文档）
│   ├── docs/             // Swagger 接口文档生成文件
│   └── http/             // HTTP 接口定义（请求/响应结构体）
│       ├── request/      // 入参结构体（如用户注册、账单新增的请求体）
│       └── response/     // 出参结构体（统一响应格式）
├── cmd/                  // 标准布局：程序入口（核心规范，必须保留）
│   └── server/           // Web 服务入口（可扩展其他 cmd，如迁移工具、CLI）
│       └── main.go       // 原 main.go 迁移至此
├── configs/              // 标准布局：配置文件（yaml/toml，原 config 目录升级）
│   └── config.yaml       // 数据库、Redis、JWT 等配置
├── internal/             // 标准布局：内部代码（不对外暴露，核心业务逻辑）
│   ├── controller/       // Gin 控制器（处理 HTTP 请求，调用 service）
│   ├── middleware/       // Gin 中间件（JWT、日志、跨域）
│   ├── model/            // 数据模型（MySQL 实体、Redis 缓存模型）
│   ├── repository/       // 数据访问层（替代原直接用 gorm，更符合 DDD 思想）
│   │   ├── mysql/        // MySQL 操作（用户、账单等 CURD）
│   │   └── redis/        // Redis 操作（缓存、黑名单）
│   ├── service/          // 业务逻辑层（核心逻辑，调用 repository）
│   └── util/             // 内部工具类（JWT、日志、加密，不对外暴露）
├── pkg/                  // 标准布局：公共工具（可对外复用，如通用响应、Redis 客户端）
│   ├── logger/           // 日志组件（zap 封装）
│   ├── redis/            // Redis 客户端封装
│   ├── jwt/              // JWT 工具封装
│   └── response/         // 统一响应格式封装
├── scripts/              // 标准布局：脚本（部署、数据库迁移、启动脚本）
│   ├── deploy.sh         // 部署脚本
│   └── migrate.sql       // 数据库表结构脚本
├── static/               // 标准布局：静态资源（如 Swagger UI 静态文件）
├── deploy/               // 部署说明（Dockerfile、docker-compose.yml、部署文档）
│   ├── Dockerfile
│   ├── docker-compose.yml
│   └── deploy.md         // 部署说明文档
├── go.mod
├── go.sum
└── README.md             // 项目说明

```
基础层→核心层→业务层→扩展层」逐步推进

阶段 1：基础环境与核心工具封装（2-3 天）
目标：搭建项目骨架，封装通用工具，验证基础环境连通性，这是所有业务的基础。
- [x] 已完成

| 步骤 | 开发内容 | 测试方式 |
|:----:|------|------|
| **1.1** | **初始化项目**<br>- 创建 go.mod，引入核心依赖（gin、gorm、go-redis、zap、jwt、viper、gin-swagger）<br>- 按既定结构创建所有空目录 | 执行 `go mod tidy`，确认依赖下载成功 |
| **1.2** | **配置管理（configs/ + pkg/）**<br>- 编写 `configs/config.yaml`（MySQL、Redis、JWT、日志配置）<br>- 封装 viper 读取配置（`pkg/config/config.go`） | 编写单元测试，验证配置能正确读取（如打印 MySQL 地址） |
| **1.3** | **日志工具封装（pkg/logger/）**<br>- 封装 zap 日志，支持控制台 / 文件输出，区分不同日志级别 | 测试日志能否正常写入文件，控制台输出格式是否正确 |
| **1.4** | **数据库 / 缓存客户端封装（pkg/）**<br>- 封装 MySQL 连接（`pkg/mysql/mysql.go`），验证连接池<br>- 封装 Redis 客户端（`pkg/redis/redis.go`），验证 ping 连通性 | 单元测试：测试 MySQL/Redis 能否正常连接，失败时抛出明确错误 |
| **1.5** | **通用工具封装（pkg/）**<br>- 统一响应格式（`pkg/response/response.go`：成功 / 失败 / 分页响应）<br>- 密码加密（`pkg/crypto/crypto.go`：bcrypt 加密 / 验证） | 单元测试：验证密码加密后能正确验证，响应格式输出符合预期 |


阶段 2：核心鉴权模块（1-2 天）
目标：完成 JWT 鉴权核心逻辑，能独立测试「生成 token→验证 token→注销 token」全流程。
- [x] 已完成


| 步骤 | 开发内容 | 测试方式 |
|------|----------|----------|
|**2.1**| JWT 工具封装（pkg/jwt/jwt.go）<br>- 实现 token 生成、验证、解析 claims<br>- 实现 token 黑名单（Redis 存储） | 单元测试：<br>1. 生成 token 后能正确解析出 UserID<br>2. 注销 token 后，验证接口返回「token 已失效」 |
| **2.2**| 鉴权中间件（internal/middleware/jwt.go）<br>- 从 Header 获取 token，验证有效性，将 UserID 存入 Gin Context | 集成测试：<br>1. 无 token 请求 → 返回 401<br>2. 无效 token → 返回 401<br>3. 有效 token → 能从 Context 获取 UserID |
| **2.3** | 日志中间件（internal/middleware/logger.go）<br>- 集成 zap 日志，记录请求方法、路径、状态码、耗时 | 测试任意请求，查看日志文件是否记录完整请求信息 |

阶段 3：用户模块（核心业务第一层，2-3 天）
目标：完成用户的 CRUD + 缓存，能独立测试「注册→登录→获取用户信息→更新用户信息」全流程。
- [x] 已完成

| 步骤  | 开发内容                                                                 | 测试方式                                                                 |
|-------|--------------------------------------------------------------------------|--------------------------------------------------------------------------|
| 3.1   | 数据模型（internal/model/user.go）- 定义 User 结构体，gorm 标签（主键、唯一索引、字段约束） | 无独立测试，后续结合 repository 测试                                     |
| 3.2   | 数据访问层（internal/repository/mysql/user_repo.go）- 实现用户注册、查询（按 ID / 用户名）、更新、删除的 CURD | 单元测试：1. 注册用户 → MySQL 能查到数据<br>2. 按用户名查询 → 能返回正确用户<br>3. 更新用户信息 → MySQL 数据同步更新 |
| 3.3   | 缓存层（internal/repository/redis/user_cache.go）- 实现用户信息缓存（写入 / 读取 / 删除） | 单元测试：1. 写入缓存后，能从 Redis 读取到用户信息<br>2. 删除缓存后，Redis 中无该键 |
| 3.4   | 业务逻辑层（internal/service/user_service.go）- 整合 repo + cache：获取用户优先查缓存，更新用户同步更新缓存<br>- 实现注册（密码加密）、登录（验证密码 + 生成 token）逻辑 | 单元测试：1. 登录时密码错误 → 返回错误<br>2. 登录成功 → 返回有效 token<br>3. 获取用户信息 → 首次查 DB，二次查缓存 |
| 3.5   | 控制器（internal/controller/user_controller.go）- 编写注册、登录、获取用户信息、更新用户信息的接口<br>- 入参校验（gin binding），调用 service 层，返回统一响应 | 接口测试（Postman/Curl）：1. POST /api/v1/user/register → 注册成功<br>2. POST /api/v1/user/login → 返回 token<br>3. GET /api/v1/user/info → 带 token 请求返回用户信息 |
| 3.6   | 路由注册（cmd/server/main.go + router/router.go）- 注册用户模块路由，登录 / 注册接口无需鉴权，其他需 JWT 中间件 | 接口测试：1. 不带 token 访问 /api/v1/user/info → 401<br>2. 带有效 token 访问 → 200 |


阶段 4：核心业务模块（账单 + 分类 + 预算，3-5 天）
目标：按「模型→repo→service→controller→路由」的顺序，逐个完成核心业务，每完成一个模块就测试。

| 模块       | 开发重点                                                                 | 测试方式                                                                 |
|------------|--------------------------------------------------------------------------|--------------------------------------------------------------------------|
| 4.1 账单模块 | - 模型（internal/model/bill.go）：金额、类型（收入 / 支出）、分类 ID、备注等<br>- repo：账单 CURD、按时间 / 分类筛选、批量删除<br>- service：账单新增时关联分类，删除账单同步清理统计缓存<br>- controller：账单增删改查、筛选、批量操作接口 | 接口测试：1. 新增账单 → MySQL 写入数据<br>2. 按分类筛选账单 → 返回符合条件的结果<br>3. 批量删除账单 → 多条数据被删除 |
| 4.2 分类模块 | - 模型（internal/model/category.go）：分类名称、类型（收入 / 支出）、用户 ID<br>- repo：分类 CURD、按用户 + 类型筛选<br>- service：分类删除前校验是否有账单关联<br>- controller：分类增删改查接口 | 接口测试：1. 删除关联账单的分类 → 返回错误<br>2. 按类型筛选分类 → 返回符合条件的结果 |
| 4.3 预算模块 | - 模型（internal/model/budget.go）：月度预算、用户 ID、已支出金额<br>- repo：预算 CURD、按用户 + 月份查询<br>- service：新增账单时更新已支出金额，超预算时日志告警<br>- controller：预算设置、查询、更新接口 | 接口测试：1. 支出超预算 → 日志中出现告警信息<br>2. 查询月度预算 → 返回剩余金额（预算 - 已支出） |

阶段 5：统计报表模块（扩展业务，1-2 天）
目标：实现高耗时的统计逻辑，重点测试缓存优化效果。

| 步骤  | 开发内容                                                                 | 测试方式                                                                 |
|-------|--------------------------------------------------------------------------|--------------------------------------------------------------------------|
| 5.1   | 业务逻辑层（internal/service/stat_service.go）<br>- 实现按周 / 月 / 年统计收支、分类占比<br>- 统计结果缓存到 Redis（过期时间 1h），避免重复计算 | 单元测试：1. 首次统计 → 计算耗时较长（查 DB）<br>2. 1h 内重复统计 → 耗时极短（查缓存）<br>3. 缓存过期后 → 重新计算并更新缓存 |
| 5.2   | 控制器（internal/controller/stat_controller.go）<br>- 编写统计报表接口（需鉴权） | 接口测试：1. 查询月度统计 → 返回收支趋势、分类占比<br>2. 不同时间维度查询 → 结果符合预期 |

阶段 7：整体联调与优化（1-2 天）
目标：全流程测试，修复边界问题，优化性能。

| 步骤  | 优化内容                                                                 | 测试方式                                                                 |
|-------|--------------------------------------------------------------------------|--------------------------------------------------------------------------|
| 7.1   | 边界场景测试<br>- 空数据查询、参数异常、并发请求、缓存失效等               | 测试：1. 无账单时统计 → 返回空数据而非报错<br>2. 并发新增账单 → 数据不重复、不丢失 |
| 7.2   | 性能优化<br>- 检查慢查询，添加 MySQL 索引<br>- 调整 Redis 缓存过期时间，优化 JWT 黑名单性能 | 压测（可选）：用 ab/jmeter 压测核心接口，确认响应时间符合预期               |



关键测试建议（边开发边测）

    单元测试优先：工具类（如 JWT、密码加密）、数据访问层（repo）必须写单元测试，保证基础层无问题；
    接口测试用 Postman/ApiPost：创建「账单管理系统」测试集合，每个接口保存请求示例，完成一个接口就跑一次；
    分层测试：先测 service 层（模拟 repo 数据），再测 controller 层（集成 Gin 上下文），最后测完整接口；
    日志辅助调试：每完成一个模块，查看日志文件，确认无错误日志，请求日志完整。

总结

    开发核心逻辑：基础工具→鉴权→用户模块→核心业务（账单 / 分类 / 预算）→统计→文档 / 部署，按「底层到上层、核心到扩展」推进；
    测试核心逻辑：每完成一个小模块就测试（如封装完 Redis 就测连通性，写完用户注册就测接口），避免问题堆积；
    关键原则：优先保证「可独立测试」，比如先让用户模块跑通，再开发账单模块，而非所有模块一起写。
